Summary

The Pattern Library is a project that we started to build a set of reusable UI components that could be consumed by applications across the company to increase consistency and speed to market. The library consists of a CSS stylesheet and JavaScript resource to reference in applications.

Project Goal: To create a single source of reusable code, allowing for developers to deliver consistent applications faster.

Team: A small group of User Experience designers and various engineers contributing to the library over the course of the project.

Role: User Experience Designer and Front-end developer

Responsibilities:

  • Redesigning the documentation page
  • Content writing
  • Developing the documentation page
  • Developing UI components
  • Project planning and prioritization

In the beginning

...there was inconsistency

As a User Experience team, we noticed inconsistency within our own team’s designs as well as in the execution by different development teams across enterprise portfolios and product lines.

Objective

Create a single repository of reusable UI components

Our goal was to create a single code source where developers could grab ‘components’ and easily insert them into their application. We decided to provide a CSS stylesheet and JavaScript resource to reference in applications across the company. Then all developers need to do is copy and paste the HTML into the application.

Approach

1. Define standards and components

Our User Experience team began defining what we considered our standards and guidelines that all application designs should follow. These included formatting of data elements (i.e. Addresses, Dates, Email, etc.), color, and typography, among other things.

We also began to identify reusable components (UI patterns that were used on multiple applications) such as Textboxes, Search Components, Modal Windows, Dropdowns, etc. Then our design team began to develop specifications for how these components should look and act.

2. Build the repository

We collaborated with a small group of developers to create the base repository for housing the components. We decided to customize the popular HTML, CSS, and JavaScript framework, Bootstrap (developed by Twitter). We thought this would help to expedite the process. To avoid excess bloat, we took bits and pieces of the Bootstrap library and customized them on a 1 by 1 basis.

3. Build the components

A small group of developers began to build components according to specifications developed by our User Experience design team.

4. Build the documentation layer (Wiki)

In order to be successful, every development library, framework, or style guide needs a great documentation site. We built a documentation ‘wiki’ where we displayed functioning components and explained how to use them.

5. Communicate the project

As this was not an official project, but a self-started collaborative project between the User Experience team and a small group of engineers, there was not a ton of visibility. We had created a great resource to improve consistency and speed up development time, but no one knew it existed.

So, we presented our solution to an audience of IT and business stakeholders and also did some hands-on training with engineers.

Uh-Oh!?!

Where's the value?

The Erie Insurance Web Asset Library was not adopted by many teams in the first year of its existence. Unfortunately, the value was not fully understood by teams.

How do we use it?

The documentation layer was a huge problem. Clear explanations were not offered and the wiki site needed to be redesigned.

What is a 'Web Asset'?

The name of the library, Web Assets, was ambiguous at best. It was unclear what the Web Asset Library was and what consumers would be getting by referencing it.

Fixing the problem

1. Research & Analysis

Our User Experience team began defining what we considered our standards and before building the initial Library, some research was done on how other companies were building UI Component Libraries, but evidently not enough. We started over and looked at some of the Pattern Libraries and Style Guides that the top companies were developing. Some of the main influencers of our refresh were: Salesforce Lightning Design System, Google Material UI, and IBM Carbon Design System.

Additionally, we renamed the ‘Web Asset Library’ to something more industry standard and something we felt would make sense to developers consuming the resource. And thus, the ‘Erie Insurance Pattern Library’ was born, or reborn.

2. Conceptual Designs

Using some industry leading style guides and pattern libraries for inspiration, we developed conceptual designs of what our ‘documentation wiki’ should look like.

3. Project management

Another opportunity we had for improvement was around project management and organization. We operated mainly in an Aglie-ish methodology, but we weren’t delivering nearly as often as we wanted to. Some of this was do to the fact that this was sort of a rogue operation, but also because we did not have an established project management plan. We created a Trello board to manage cards and gain visibility around work. This helped us commit to deadlines, create a backlog, and chunk our work into sprints and specific development cards.
